# Structure & legibility

_Repo layout, modularity, config, file size, import boundaries, monorepos_

**Overview:** An LLM evaluator localizes code before it reasons about it: published coding agents (Agentless, LocAgent, RepoGraph) run a name-only scan of the file/directory tree as their first step, then BM25/grep over fully-qualified names, then traverse the import graph — so a repo whose structure, names, and imports make the right file cheaply findable scores higher because the agent actually reaches the relevant code, and lower when it has to fall back to lossy embeddings or load whole subsystems. Legibility also compounds with correctness: problem-solving accuracy degrades as a function of raw context length itself, so small cohesive files and flat control flow keep the working set inside the model's high-attention region. Critically, the rules that genuinely help an agent (descriptive paths, one consolidated pyproject.toml, mirrored tests/, enforced import boundaries, complete .env.example) are the same ones expert human reviewers already value, so they raise ratings for the right reasons. The same surface is, however, exploitable by documented LLM-judge biases (authority/self-correctness comments, verbose docs, illusory complexity), so the section pairs each genuine rule with the gaming move it must not be confused with.

## Rules

### 1. Make the right file findable by NAME alone: name every file and directory after the domain concept it implements, putting the locating keyword in the path (auth/password_validator.py, payment-repository.py), and reserve generic buckets (utils.py, helpers.js, manager.py, misc.py, data2/) for never.  _[strong]_
- **Why:** The first localization step a coding agent performs operates on the file/directory tree as plain text, before any code is read. Agentless reports the ground-truth file is in the top candidates for 77.7% of SWE-bench cases from this name-only step, narrowing scope from ~3,305 to ~246 LOC. Generic names carry zero signal to rank on, and a name matching everything (grep utils → whole repo) is as useless as one matching nothing. Descriptive paths are also a long-standing human readability value, so this is genuine quality, not a judge artifact.
- **How:** Use [feature]-[type].[ext] (user-service.ts, auth-middleware.py) and group into domain directories (src/users/, src/payments/) so glob('**/*{auth,user}*') and grep hits cluster by directory. Qualify any generic suffix (email_utils.py, RetryManager, not utils.py/Manager). If a name must stay non-obvious, add a docstring/header line with the searchable plain-language term. Optionally machine-enforce with eslint-plugin-check-file (filename-naming-convention) for JS/TS or a pre-commit hook for Python (Ruff has no native file-name rule).
- **Example:** Good: osfl/store.py exposes class Store with load/save/get/list/upsert/delete — verbs a CRUD query greps for. Bad: a def attempt_loop() that should be def retry_with_backoff(), or service.ts that should be user-service.ts.
- **Sources:** https://arxiv.org/html/2407.01489v1, https://arxiv.org/pdf/2503.09089, https://yage.ai/share/why-coding-agents-still-use-grep-en-20260327.html, https://github.com/dukeluo/eslint-plugin-check-file

### 2. Use ONE term per domain concept and ONE verb family across the whole call chain; align internal names with external/issue-tracker vocabulary.  _[strong]_
- **Why:** Name drift (process/handle/execute the same Payment; get/fetch/retrieve/load the same User; userService.js vs user_service.py vs UserService.ts) fragments grep/BM25 results and forces a lossy embedding fallback, leaving a semantic gap between the query and the code. Consistency keeps a single literal search landing the intended file and is independently a human code-quality value.
- **How:** Pick the canonical verb for each CRUD operation and the canonical noun for each entity, then apply them everywhere. If the API/spec says 'webhook', name everything webhook_* (not callback_*, hook_*). Keep one casing convention per language. Spell names out (verb+noun for functions: fetch_user_by_id; noun for classes/files) so the keyword a query uses literally appears in the entity's name.
- **Example:** Pick fetch_user / create_user / update_user / delete_user and never introduce get_user or load_user as synonyms in another module.
- **Sources:** https://arxiv.org/pdf/2407.01489, https://aipatternbook.com/naming

### 3. Choose Python layout deliberately and verify import isolation — do NOT treat the literal 'src/' token as a quality score.  _[strong]_
- **Why:** src-layout (importable code in src/<pkg>/, tests/ and docs/ as siblings) is strongly suggested (pytest, pyOpenSci, the PyPA tutorial — not a blanket PyPA mandate) because Python puts CWD first on sys.path, so a flat package at root can be imported instead of the installed copy, masking packaging errors and letting tests pass against working-directory files that would never ship. BUT flat layout is a defensible, expert choice: NumPy, SciPy, and Matplotlib deliberately use it. The decisive quality signal is that imports resolve to the INSTALLED package and a correct [build-system] exists — both achievable flat or src. Rewarding the 'src/' string is a false-positive heuristic to avoid.
- **How:** For new distributable packages prefer src-layout and develop with an editable install (uv pip install -e . / pip install -e .). If flat, ensure the package is pip-installed before tests run and that [build-system] is valid. Either way, tests/ and docs/ live at the repo root as siblings of the code, NOT inside the package, and are excluded from the shipped wheel.
- **Example:** This repo is legitimately flat: osfl/ at root with a valid pyproject.toml ([build-system] requires=['hatchling']; [tool.hatch.build.targets.wheel] packages=['osfl']). That is correct, not a defect, because the package is installed before tests and the build backend is configured.
- **Sources:** https://packaging.python.org/en/latest/discussions/src-layout-vs-flat-layout/, https://docs.pytest.org/en/stable/explanation/goodpractices.html, https://www.pyopensci.org/python-package-guide/package-structure-code/python-package-structure.html

### 4. Put a predictable, canonically-named file set at the repo root and consolidate ALL Python config into one pyproject.toml.  _[strong]_
- **Why:** Canonical names/locations (README.md, LICENSE, pyproject.toml, top-level tests/ and docs/, plus CHANGELOG.md/CONTRIBUTING.md for published packages) let a reviewer or agent locate build, license, and contribution info without searching. Scattering config across setup.py + setup.cfg + tox.ini + .flake8 produces a real defect class — duplicated and contradictory settings; the modern toolchain (ruff, mypy, pytest, build backends) all read pyproject tables, so one file removes that ambiguity.
- **How:** Use [build-system] for the backend, [project] for metadata, and tool tables ([tool.ruff], [tool.pytest.ini_options] with testpaths, [tool.mypy]). Anchor test discovery with testpaths and, for new projects, prefer pytest's importlib import mode. Drop setup.py unless a build hook genuinely needs it. Keep tests/ out of the application package and out of the wheel.
- **Example:** [tool.pytest.ini_options]\ntestpaths = [\"tests\"]\naddopts = [\"--import-mode=importlib\"]  — with tests/test_foo.py mirroring osfl/foo.py so the test for a module is deterministically locatable.
- **Sources:** https://docs.pytest.org/en/stable/reference/customize.html, https://docs.pytest.org/en/stable/explanation/goodpractices.html, https://www.pyopensci.org/python-package-guide/package-structure-code/python-package-structure.html

### 5. Keep the dependency graph explicit and traversable, with one clear public entry point per module.  _[strong]_
- **Why:** Imports at module top, explicit named imports (no wildcard/dynamic/reflection-based imports), no hidden coupling via globals or monkey-patching, and a curated public surface are exactly the contain/import/invoke edges agents multi-hop along. LocAgent's graph-guided localization beats the best embedding baseline by +9.49pp at file level (94.16% vs 84.67% Acc@5) and +18.61pp at function level; RepoGraph reports 32.8% average relative improvement on SWE-bench-Lite. Retrieval baselines fail at function level even when the target function is named in the query. Explicit imports are also grep-able and refactor-safe — a human value.
- **How:** Declare a curated public API per module (Python __all__, Node exports map). Forbid star/dynamic imports. Open each module file with a 2-3 sentence docstring stating what it is and its key convention — it is read on the first cat/grep hit and converts found-by-name into understood. Optionally ship a generated import/dependency graph for large repos.
- **Example:** osfl/engine/__init__.py opens with 'OSFL math engine — pure numpy, no I/O, deterministic given a seed.' and re-exports a curated surface (beta_from_mean, simulate_funnel, validate_strategy, ...) instead of leaking every internal symbol.
- **Sources:** https://arxiv.org/html/2503.09089v1, https://arxiv.org/abs/2410.14684

### 6. Enforce module/dependency boundaries as a machine-checkable CI gate, never as prose alone, and ban cyclic dependencies entirely.  _[strong]_
- **Why:** Lint and tests provably do NOT enforce dependency direction, so an architecture described only in a README or AGENTS.md can be silently violated by the real import graph — which is itself a top judge-gaming vector (claiming modularity the imports break). A green CI run on an enforced contract is a falsifiable artifact that makes the graph the agent reasons over actually true.
- **How:** Python/uv: add a .importlinter declaring a layers contract (e.g. app > orchestrator > agents > engine > models), forbidden contracts to ban specific edges, and independence contracts to forbid sibling cycles; run lint-imports in CI (non-zero exit on violation). JS/TS monorepos: Nx @nx/enforce-module-boundaries with depConstraints, or eslint-plugin-boundaries; pnpm's strict node_modules additionally makes undeclared (phantom) imports impossible. Documentation must MIRROR enforcement, never substitute for it.
- **Example:** [importlinter:contract:layers]\ntype = layers\nlayers =\n    osfl.app\n    osfl.orchestrator\n    osfl.agents\n    osfl.engine\n    osfl.models — the OSFL tree already satisfies this: models.py imports zero siblings (leaf) and osfl/engine/*.py never imports agents/orchestrator/app.
- **Sources:** https://import-linter.readthedocs.io/en/latest/contract_types.html, https://nx.dev/docs/features/enforce-module-boundaries

### 7. Organize by feature/vertical slice and budget coupling/cohesion explicitly; use the change-together test for boundaries.  _[strong]_
- **Why:** Collocating a feature's logic, types, schemas, and tests in one narrow directory means a single semantic query prunes whole unrelated subsystems, minimizing the context an agent must load (token economy + retrieval recall). CBO (coupling-between-objects) and LCOM (lack-of-cohesion) are among the strongest internal predictors of change-proneness (Spearman ~0.51 / ~0.49 across 22 Apache/Eclipse systems) — present them as maintenance/change predictors, not bug predictors. The co-change criterion is falsifiable from git history: files that always change together but live in different modules signal a missed boundary; a God class whose parts change for unrelated reasons signals a needed split.
- **How:** Break monolithic Service/Repository god-classes into single-purpose files (feature/subsystem/{index, utils, index.test, utils.test}). Add a coupling/complexity metric to CI (radon cc/mi, lizard for CCN, SonarQube/Understand for CBO+LCOM) as a hygiene floor, and split units with high CBO or LCOM. Operationalize boundaries with git log --name-only co-change frequency rather than taste. Pair any line-count limit with a single-responsibility check — do not over-fragment into hundreds of trivial 1-5 line files.
- **Example:** billing/refund/refund.py + billing/refund/refund_test.py (collocated) instead of scattering across controllers/billing.py, services/billing.py, models/billing.py which forces loading three directories to assemble one feature.
- **Sources:** https://arxiv.org/pdf/2310.03673, https://arxiv.org/pdf/2411.01012, https://dev.to/somedood/coding-agents-as-a-first-class-consideration-in-project-structures-2a6b

### 8. Keep the unit an LLM must reason over small and cohesive (roughly under ~500 lines), and split on AST boundaries, never line/token boundaries.  _[strong]_
- **Why:** Problem-solving accuracy degrades as a function of total context LENGTH itself — independent of retrieval quality or needle position (degradation persists even when distractors are masked and evidence sits immediately before the question). Successful agentic SWE-bench trajectories stay under ~20-30K tokens; a single-pass 64K injection collapses some models to near 0% even with 100% file recall (failure modes: hallucinated diffs, wrong line numbers). 1000+ line god-files are a comprehension liability, not just a style issue. Keeping a function's signature, body, and docstring together as one self-contained unit is genuine cohesion that humans and refactoring tools value.
- **How:** Use tree-sitter / AST-aware chunking for any RAG/embedding index; never split a single routine across chunks or files. Front-load load-bearing context at the top of each file (module docstring, key types, public API) so nothing critical lands in the low-attention 'middle' (U-shaped attention). Treat any line limit as a warn-level nudge that counts real code only, never a hard quality gate. Do large multi-file refactors in small batches so a changed signature and all its call sites stay in the window together.
- **Example:** max-lines as a soft signal: 'max-lines': ['warn', { max: 400, skipBlankLines: true, skipComments: true }] paired with 'max-lines-per-function', with a message telling the agent to split by single responsibility.
- **Sources:** https://arxiv.org/abs/2510.05381, https://arxiv.org/abs/2602.16069, https://arxiv.org/abs/2506.19897, https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents

### 9. Make code LLM-legible by reducing nesting depth and flattening control flow, not by chasing a low cyclomatic-complexity number.  _[strong]_
- **Why:** Once length is controlled for, classical metrics (cyclomatic complexity, Halstead, maintainability index) do NOT predict LLM difficulty; compositional/nesting depth and branching density do. Flat control flow with early returns and guard clauses is also a textbook human-reviewer preference, so this corrects a common metric-chasing anti-pattern while improving genuine readability. Treat lint/smell counts as a hygiene floor, not a quality ceiling (SonarQube issues have a statistically significant but negligible effect on faults, Cliff's delta -0.06).
- **How:** Refactor nested if/else into guard clauses and extracted, well-named helpers. Keep each function short enough that its whole logic sits in the high-attention head/tail region. Prioritize the few Reliability/Bug lint rules that map to real faults over broad Code Smell rules.
- **Example:** def charge(user):\n    if not user.active: return Err('inactive')\n    if user.balance < price: return Err('insufficient')\n    return do_charge(user) — identical cyclomatic complexity to a nested version, but every branch stays in the model's high-attention region.
- **Sources:** https://arxiv.org/abs/2602.07882, https://arxiv.org/abs/1908.11590

### 10. Centralize config in ONE typed settings module and commit a COMPLETE .env.example; keep config separated from code (12-Factor).  _[strong]_
- **Why:** Scattered os.environ.get()/os.getenv() reads let missing or mistyped values fail deep in a request path instead of at startup. A single typed object (pydantic-settings BaseSettings) that declares every variable's type, default, and required-ness and validates on import makes misconfiguration fail fast with a named error — and arXiv 2412.11121 ties this class of error to the two largest production-failure categories (Resource Unavailability 35.23%, Constraint Violation 22.15%). Any env var the code reads but the example omits is a silent-failure landmine. These are mechanically verifiable, human-valued moves, not judge artifacts.
- **How:** Replace every scattered read with one config.py (BaseSettings, frozen). Type secrets as SecretStr and read via .get_secret_value(); never log the whole settings object. Document each non-obvious setting's effect/units/range in the field docstring and mirror it as a one-line comment in .env.example. Gitignore the real .env with the negation pattern (.env, .env.*, !.env.example). Apply the litmus: 'could this repo go open-source right now without leaking a credential?' Never branch on environment names (if ENV=='prod').
- **Example:** This repo's litmus PASSES (.gitignore has .env + .env.* + !.env.example; OPENAI_API_KEY= is blank), but it has the incomplete-example landmine: osfl/app.py:38 reads OSFL_STORE_PATH yet .env.example never lists it. Fix by adding '# Path to the JSON store (default: data/store.json). Optional.\nOSFL_STORE_PATH=' and folding the three reads (llm.py:32-33, app.py:38) into one osfl/config.py.
- **Sources:** https://docs.pydantic.dev/latest/api/config/, https://12factor.net/config, https://arxiv.org/html/2412.11121

### 11. Adopt explicit workspace layout for monorepos and choose mono-vs-multi conditionally — never assume 'monorepo = more legible'.  _[moderate]_
- **Why:** Turborepo/pnpm conventions give an agent a real dependency manifest instead of forcing it to infer architecture from relative-path spaghetti. A ../ import crossing a package boundary is a greppable refactor signal an evaluator can check mechanically. But navigation difficulty grows with repo size (Brito/Terra/Valente MLR 2018): a large, unstandardized, untooled monorepo is LESS navigable than well-bounded multi-repo. private:true + a committed lockfile + a pinned packageManager are real correctness signals (prevent accidental publish, ensure reproducible installs).
- **How:** Split apps/* (deployables) and packages/* (shared libs), each with its own named manifest and single-level globs (no apps/**). Set root package.json private:true, pin packageManager (pnpm@x.y.z), commit one lockfile, use scoped names (@org/pkg) and prefer exports over main. Import siblings via their public exports (install as a dependency), never via ../. Python: use uv workspaces (one root uv.lock, each member's pyproject.toml, [tool.uv.sources] {workspace=true}) — but Astral's own guidance says DON'T use a workspace when members have conflicting requirements; use path deps or separate projects instead. Choose monorepo only when cross-package changes are frequent AND you invest in build-graph/affected-detection/boundary-lint tooling.
- **Example:** In apps/web/package.json: \"@acme/ui\": \"workspace:*\"; import { Button } from '@acme/ui' — NEVER from '../../packages/ui/src/button'. For Node API boundaries, an exports map (\".\": \"./dist/index.js\") makes pkg/internal/secret.js throw ERR_PACKAGE_PATH_NOT_EXPORTED for package-name imports (an API-boundary convention, not a hard security boundary — a direct require of an absolute subpath still loads it).
- **Sources:** https://turborepo.dev/docs/crafting-your-repository/structuring-a-repository, https://arxiv.org/abs/1810.09477, https://nodejs.org/api/packages.html

### 12. Add a terse root AGENTS.md (mirrored to CLAUDE.md) containing ONLY non-inferable content, and surface the architecture only when it is enforced and accurate.  _[moderate]_
- **Why:** AGENTS.md is an emerging cross-tool convention (Claude Code reads it as a CLAUDE.md fallback; nested files are nearest-wins). Its value is exact build/test/lint commands, non-standard tooling (uv vs pip), and counterintuitive patterns with their reason — content an agent cannot infer. A navigation/architecture map helps ONLY if it matches the real directory/dependency graph; a stale or generic map actively misleads an agent and is a prime judge-gaming vector. Restating the README or padding with generic 'Project Structure' prose adds tokens, worsens context rot, and (per practitioner findings) can lower task success while raising cost.
- **How:** Put executable commands early in code blocks (uv sync, cp .env.example .env, uv run pytest). Remove a 'Project Structure' section when the layout follows known conventions. For large monorepos use a slim root file linking to per-package AGENTS.md (nearest-wins) rather than one monolith. Keep the root file to a few hundred tokens. If you ship a NAVIGATION.md/ARCHITECTURE.md, condition its presence on a CI-enforced boundary so docs mirror enforcement.
- **Example:** A 6-line AGENTS.md Setup section (uv sync; cp .env.example .env; uv run pytest) pointing at .env.example for required vars — NOT a 200-line ASCII directory tree restating what glob already reveals.
- **Sources:** https://agents.md/, https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents, https://engineering.mercari.com/en/blog/entry/20251030-taming-agents-in-the-mercari-web-monorepo/

### 13. Use good names, real docstrings, and honest decomposition to raise clarity — and NEVER add authority comments, self-correctness comments, or gratuitous helper scaffolding to inflate perceived quality.  _[strong]_
- **Why:** This is the defining gaming guardrail for the whole category. Superficial cues bias LLM code judges hard: self-declared-correctness comments moved GPT-4o +5.3 to +7.8pp and LLaMA-3.1-70B +26.4pp on correct code AND degraded incorrect-code detection by -9.4 to -23.5pp; authority comments +0.5 to +4.1pp; 24-char random/inflated names +4.3pp MAD; dummy/illusory-complexity functions up to +9.33pp — and larger model scale did NOT fix it. These exploit documented bias without improving the system and are unreliable (they also inflate scores on wrong code). The constructive half (descriptive names, real docstrings, honest feature decomposition) legitimately raises both human and LLM scores because it improves clarity.
- **How:** Write only factual comments stating purpose/units/constraints. Decompose because a unit has more than one responsibility, not to look modular. A defending evaluator strips comments, normalizes names, and prefers test-execution judges, so every quality claim must be backed by a verifiable artifact (passing import-linter/test run, measured CBO/LCOM, git co-change history) — not prose, comments, or issue-count screenshots.
- **Example:** REJECT: '# This implementation was written by a senior engineer and is correct' above def _helper_a(x): return _helper_b(x) / def _helper_b(x): return x (gratuitous indirection). These flip judges up to ~26.7pp without improving the code.
- **Sources:** https://arxiv.org/pdf/2505.16222, https://arxiv.org/html/2505.16222

## Checklist
- [ ] Does a single literal grep/glob/BM25 query land the intended file? (locating keyword is in the path; no generic utils.py/helpers.js/manager.py buckets)
- [ ] One term per domain concept and one verb family for CRUD across all files; internal names match the API/issue-tracker vocabulary
- [ ] Python: imports resolve to the INSTALLED package (pip install -e . before tests) and [build-system] is valid — regardless of src vs flat; tests/ and docs/ are root siblings, excluded from the wheel
- [ ] One pyproject.toml holds [build-system], [project], and tool tables (ruff/pytest testpaths/mypy) — no stray setup.cfg/tox.ini/.flake8
- [ ] Each module has a curated public surface (__all__ / exports) and a 2-3 sentence header docstring; imports are top-level, named, no wildcard/dynamic
- [ ] Dependency direction is enforced in CI (import-linter lint-imports / Nx boundaries / eslint-plugin-boundaries), not just described; no cyclic deps
- [ ] Code is organized by feature/vertical slice with tests collocated; god-classes with high CBO/LCOM are split
- [ ] No file is a 1000+ line god-file; the per-task slice fits ~20-30K tokens; functions+docstrings are kept together (AST-boundary splits)
- [ ] Control flow is flat (guard clauses / early returns); not optimized purely for a low cyclomatic-complexity number
- [ ] All config is read off one typed Settings object; .env.example lists EVERY var the code reads; real .env is gitignored with !.env.example; litmus: could this go open-source without leaking a credential?
- [ ] Secrets typed as SecretStr; the whole settings object is never logged; no branching on ENV name
- [ ] Monorepo: apps/* + packages/* with named manifests, private root, pinned packageManager, committed lockfile; no ../ cross-package imports
- [ ] AGENTS.md (mirrored to CLAUDE.md) is terse, commands-first, non-inferable only; any architecture map matches the real graph and is CI-enforced
- [ ] No authority comments, self-correctness comments, inflated names, or dummy/illusory-complexity helpers; every quality claim is backed by a runnable artifact

## Anti-patterns
- Token-rewarding the literal 'src/' string: flat layout is a legitimate expert choice (NumPy, SciPy, Matplotlib use it). The real signal is installed-package import isolation + a valid [build-system], not the directory name.
- Generic catch-all names (utils.py, helpers.js, manager.py, service.ts, handleStuff, misc, data2, temp) that match dozens of unrelated files — as useless to an agent as a name that matches nothing.
- Name drift / synonym soup: mixing process/handle/execute for one Payment, get/fetch/retrieve for one User, or userService.js vs user_service.py vs UserService.ts — fragments grep/BM25 and forces lossy embedding fallback.
- Architecture-as-prose: an elaborate README/ARCHITECTURE.md/AGENTS.md describing layers that the actual import graph violates. Documentation must mirror enforcement, never substitute for it.
- Verbose padding to fake thoroughness: ASCII directory-tree dumps of a conventional layout, decorative badges, per-folder placeholder READMEs, long INDEX.md/AGENTS.md full of vague descriptions ('overview of pricing', 'core utilities') — inflates tokens and context rot without improving any grep/glob hit.
- Over-splitting: shattering code into hundreds of trivial 1-5 line files or deep nesting just to satisfy a raw line-count limit — trades file size for navigation cost (more glob/grep/read calls, fragmented logic).
- Treating a line-count number (150/200/500) as the sole quality signal while a 200-line file still mixes three responsibilities; or chasing a green SonarQube/lint score (negligible real fault effect, Cliff's delta -0.06) as if it were a quality ceiling.
- Scattered ad-hoc os.environ.get()/os.getenv() reads and an incomplete .env.example that omits a var the code actually reads (e.g. OSFL_STORE_PATH here) — silent-failure landmines that fail deep in a request path instead of at startup.
- Dumping a 64K/200K context 'because the window allows it': accuracy degrades with raw length, and single-pass mega-injections collapse some models to near 0% even with full file recall.
- BIAS-GAMING (explicitly avoid): self-declared-correctness comments (+up to ~26.7pp on judges, and they inflate scores on WRONG code), authority comments ('reviewed by a senior SRE'), inflated/random long identifier names, dummy functions and illusory-complexity scaffolding, and self-affirming config banners ('# production-grade, secure, validated config'). Larger model scale does not fix these; a defending evaluator strips them.
