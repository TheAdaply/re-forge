# Tech stack & ecosystem

_Language choice, framework idioms, dependency selection, Python(uv/ruff) & Node/TS specifics_

**Overview:** LLM evaluators score a project partly on genuine correctness signals (tests pass, types check, no dead code) and partly on familiarity: code that matches dominant training-data patterns is lower-perplexity and gets rated higher, often beyond its real quality gap. The two overlap heavily — a high-resource language, canonical framework layout, idiomatic naming, and a committed linter/formatter config are simultaneously what expert humans value AND what looks "low-surprise" to a judge. The durable strategy is to invest in the levers that survive execution-based and rubric-based review (mainstream stack, runnable quality gates, real tests/types, maintained deps), because single-run judge scores are unreliable (~50-60% consistency, position bias can swing accuracy ~75%→38%) and any rating bought purely by presentation tricks (verbosity, authority comments, "v2/refined" labels) collapses under a careful reviewer who strips labels, swaps positions, and runs the code.

## Rules

### 1. Default to a high-resource language (Python, TypeScript/JavaScript, or Java) for any greenfield project you want an LLM to build, evaluate, and maintain well. Avoid low-resource languages (OCaml, Racket, R, Julia, Lua, Haskell) and bespoke DSLs unless the domain requires them.  _[strong]_
- **Why:** Corpus proportion caps BOTH generation and evaluation quality. MultiPL-E/MultiPL-T (StarCoderBase-15B) shows pass@1 of ~30.6 for Python vs ~6.9 for OCaml, with a ~20x gap at 1B, attributed directly to corpus size. This is genuine, not just judge bias: tool maturity, ecosystem docs, hiring pool, and human-reviewer familiarity all independently favor mainstream languages.
- **How:** Pick Python/TS/Java for the core. Only reach for a niche language when a domain constraint (e.g. proof assistants, embedded, data-science DSL) genuinely forces it. If forced, apply the niche-stack mitigation rule below.
- **Example:** OSFL is correctly Python 3.12 (requires-python = ">=3.12") — a high-resource choice.
- **Sources:** https://arxiv.org/abs/2308.09895, https://arxiv.org/abs/2503.17181

### 2. Write idiomatic, canonical-layout code that matches the framework's official docs verbatim: standard-library idioms over clever one-liners, the conventional project structure (Django myapp/models.py, views.py, urls.py; Next.js app/ router with page.tsx/layout.tsx), and names that match the docs. Avoid bespoke abstractions the model has rarely seen.  _[strong]_
- **Why:** LLM judges rate lower-perplexity text/code higher independent of authorship (Self-Preference Bias, 2410.21819), and more strongly than humans do. The same behavior written in a 'surprising' (higher-perplexity) way scores lower. This is also genuine quality — canonical layout and idioms are more maintainable and onboardable for humans, and an agent/judge can locate components without reading config.
- **How:** Follow convention-over-configuration: use the framework's default directory layout, routing, and entrypoints (Rails resource conventions, Spring Boot starters/autoconfiguration, NestJS modules). Do not hand-roll bespoke config where a convention exists. Split a 2000-line app.py into the canonical module layout.
- **Example:** Django app as myapp/models.py + myapp/views.py + myapp/urls.py (not one monolithic app.py); Next.js using app/page.tsx + app/layout.tsx matching official naming.
- **Sources:** https://arxiv.org/abs/2410.21819, https://en.wikipedia.org/wiki/Convention_over_configuration

### 3. Run an auto-formatter + linter with config committed to the repo and enforced in CI and pre-commit — never via CLAUDE.md prose asking the LLM to do style work. Python: Ruff (lint+format+import-sort). JS/TS: Prettier for formatting + ESLint for correctness only. Rust: rustfmt/clippy. Go: gofmt/golangci-lint.  _[strong]_
- **Why:** Mechanical enforcement is genuine, expert-valued quality that is non-gameable and verifiable: it catches real bugs, guarantees one consistent convention, and makes diffs show only meaningful changes. It survives test/rubric evaluation regardless of any judge. Ruff is a drop-in Black replacement (>99.9% identical output) consolidating ~40 tools. typescript-eslint docs state verbatim to use Prettier for formatting, not ESLint formatting rules.
- **How:** Add a [tool.ruff] section to pyproject.toml with an explicit lint set; wire ruff check --fix and ruff format --check into CI and .pre-commit-config.yaml (astral-sh/ruff-pre-commit, pinned rev). For TS, keep .prettierrc and eslint.config.mjs separate; never enable ESLint formatting rules.
- **Example:** [tool.ruff]\ntarget-version = "py312"\nline-length = 100\n[tool.ruff.lint]\nselect = ["E", "F", "I", "UP", "B", "SIM", "C4", "PTH", "RET", "TC"]\n— paired with `uv run ruff check --fix .` in CI. (OSFL's pyproject.toml currently has NO [tool.ruff] section — this is a missing high-value add.)
- **Sources:** https://docs.astral.sh/ruff/faq/, https://typescript-eslint.io/users/configs/

### 4. Add a static type checker as a CI gate and annotate all public signatures and model/dataclass fields. Python: mypy --strict / pyright / ty. TypeScript: "strict": true in tsconfig.json, then go further with @tsconfig/strictest or at minimum noUncheckedIndexedAccess: true.  _[strong]_
- **Why:** Types help BOTH the rater and real correctness, so investing in them is non-gaming. A large empirical study (FSE 2022, 9,655 projects / 1.1M commits) found a type checker avoids ~15% of type-related defects; the JS 'To Type or Not to Type' study (ICSE 2017) measured ~15% of public bugs caught by static typing. noUncheckedIndexedAccess makes arr[0] return T | undefined, eliminating a whole class of 'cannot read property of undefined' crashes. A green strict type-check is a falsifiable quality gate, not an impression.
- **How:** Add [tool.mypy] strict = true to pyproject.toml and run `uv run mypy <pkg>` in CI. For new TS projects extend @tsconfig/strictest and add eslint.config.mjs extending tseslint.configs.strictTypeChecked with parserOptions.projectService:true to enable type-aware rules (no-floating-promises, no-misused-promises, no-unsafe-*) that tsc alone misses.
- **Example:** [tool.mypy]\nstrict = true\n(OSFL's pyproject.toml has pytest config + hatchling but NO [tool.mypy] section.)  TS: { "extends": "@tsconfig/strictest/tsconfig.json", "compilerOptions": { "module": "NodeNext", "noUncheckedIndexedAccess": true } }
- **Sources:** https://software-lab.org/publications/fse2022_type_study.pdf, https://earlbarr.com/publications/typestudy.pdf, https://github.com/tsconfig/bases/blob/main/bases/strictest.json

### 5. Ship executable evidence alongside idiomatic code: real fail-to-pass tests (keep existing pass-to-pass green, add regression tests for fixed bugs), type annotations, and a CI config — so a judge or human grounds assessment in verification rather than surface cues. Remove dead/unused functions and variables.  _[strong]_
- **Why:** Execution-based evaluation is where genuine and perceived quality align and where gaming collapses. SWE-bench+ found ~31% of 'passing' patches passed only due to weak tests and ~33% leaked solutions, so hollow tests (assert-runs-without-error) fail under any execution harness. Rubric/test-based judging reduces susceptibility to surface bias (2505.16222). Removing dead code directly defuses the 'illusory complexity' bias by deleting the flaw rather than hiding it.
- **How:** Write pytest tests that assert real behavior (avoid trivial 'function ran' assertions); add a regression test for every fixed bug. Run dead-code detection (Ruff F401/F841, deptry, depcheck) and delete what it flags. Make the test command a one-liner an evaluator can reproduce.
- **Example:** uv run pytest -x  (OSFL already uses pytest + hypothesis — the lever is test substance, not test presence).
- **Sources:** https://arxiv.org/pdf/2410.06992, https://arxiv.org/html/2505.16222

### 6. Use clear, descriptive, purpose-matching identifier names and keep comments/docs truthful to the code. Python: snake_case for functions/variables; rename cryptic names (ml/mlt) to descriptive ones (max_length/current_length). Avoid 'atoms of confusion' linters don't auto-rewrite: operator-precedence tricks, variable shadowing, implicit conversions, brace-less blocks.  _[strong]_
- **Why:** Core readability valued by all expert reviewers and robust under test/rubric evaluation. Two studies show it causally moves judge scores correctly: Simoes & Venson (9 LLMs; obscuring names produced significant downward shifts; coherence of name/comment/doc with purpose was the strongest signal) and 2506.12014 (snake_case, descriptive non-digit-suffixed names). Atoms of confusion spike token-level perplexity (11→99, max 555) and co-locate with human EEG confusion signals (rho=0.47, p<0.001) per ICSE 2026 (2508.18547). CAUTION: 'descriptive' means meaningful, NOT artificially long.
- **How:** Enable Ruff N (pep8-naming) and SIM/UP. Add parentheses around mixed operators, give shadowing variables unique names, write full statements not precedence tricks.
- **Example:** GOOD: current_length = max_length - offset.  BAD (cryptic): mlt = ml - o.  EQUALLY BAD (length-gaming): the_current_length_value_after_offset_subtraction = ... — a reviewer flags the padded form.
- **Sources:** https://arxiv.org/abs/2507.05289, https://arxiv.org/pdf/2506.12014, https://arxiv.org/html/2508.18547v1

### 7. Vet every dependency against machine-checkable maintenance signals before adding it, and prefer the ecosystem-default mainstream library over an obscure single-maintainer package — but only when a dependency is genuinely needed. Reject archived-upstream, multi-year-stale, or deprecated packages unless they implement a frozen stable spec.  _[strong]_
- **Why:** Archived/abandoned deps mean no security fixes (OpenSSF Scorecard 'Maintained': ≥1 commit/week over trailing 90 days is the top bar; archived is lowest). Popularity is a documented protective factor against package hallucination (~19.7% of LLM samples reference hallucinated packages; commercial models 5.2% vs open-source 21.7%). CAVEAT: do not over-add mainstream deps — NumPy is imported unnecessarily in up to 48% of LLM cases; expert reviewers penalize bloat and unnecessary heavy deps.
- **How:** Check the source repo is not archived, last commit within ~180 days, recent releases, >1 maintainer for critical deps. Rank candidates by release recency / update cadence, NOT GitHub stars (stars lag and mislead). Run abandonment detection in CI (Python: pip-abandoned exit code 9 on archived/Inactive upstream, deptry; Node: depcheck + npm WARN deprecated).
- **Example:** OSFL's pyproject.toml is a minimal, all-mainstream, actively-maintained set (fastapi>=0.110, uvicorn, numpy>=2.1, pydantic>=2.6, pyyaml, openai>=2.40.0) with the ROADMAP discipline 'zero new mandatory dependencies; vendor tiny snippets instead of pulling heavy frameworks (LangGraph, CrewAI, AutoGen)' — the genuine version of 'prefer mainstream, avoid bloat'.
- **Sources:** https://github.com/ossf/scorecard/blob/main/docs/checks.md, https://github.com/chris48s/pip-abandoned, https://arxiv.org/html/2503.17181v2

### 8. Commit a lockfile and add an automated dependency-update tool config — AND actually merge its PRs. Python: commit uv.lock + .github/dependabot.yml. Node: package-lock.json + dependabot/renovate. Declare your project's maturity honestly in pyproject.toml Trove classifiers.  _[strong]_
- **Why:** Pinning + automated updates is real reproducibility/security hygiene that also flips Scorecard's Dependency-Update-Tool and Pinned-Dependencies checks to high. CAVEAT: Scorecard only detects the config's presence, NOT whether PRs are merged — adding dependabot.yml but letting updates rot is theater and an explicit anti-pattern.
- **How:** Add .github/dependabot.yml (package-ecosystem: pip, schedule weekly), keep uv.lock committed, and gate CI with pip-abandoned. Set classifiers to honest maturity (e.g. 5 - Production/Stable, never a stale 4 - Beta, and never depend on a 7 - Inactive package).
- **Example:** .github/dependabot.yml:\nversion: 2\nupdates:\n  - package-ecosystem: "pip"\n    directory: "/"\n    schedule: { interval: "weekly" }\n(OSFL already commits uv.lock; it has no .github/ dir yet — dependabot + a CI workflow are the missing adds.)
- **Sources:** https://github.com/ossf/scorecard/blob/main/docs/checks.md, https://pypi.org/classifiers/

### 9. Add ONE terse, human-curated AGENTS.md at the repo root (~under 150-200 lines): exact copy-pasteable tool-prefixed commands FIRST, then exact stack versions and counterintuitive repo-specific conventions with their mechanistic 'why'. If CLAUDE.md/copilot-instructions are also needed, point them at AGENTS.md (symlink) rather than duplicating. Do NOT auto-generate it, restate framework basics, embed stale code tours, or pad with architecture prose.  _[moderate]_
- **Why:** A controlled ETH Zurich study (2602.11988) found human-written context files improved task success only ~4% and only for minimal requirements/custom commands/tooling, while auto-generated files DECREASED success ~2-3% and raised cost 20%+. Naming a tool like uv made agents use it ~2.5x more. So the only safe form is minimal and command-first; this is genuine quality (less over-instruction = better agent behavior, and reproducible verification for the rater). Codify any correction you make to the agent twice.
- **How:** Lead with setup/test/lint/typecheck commands using the exact non-default tool prefix. Add a 'never touch' constraints line. Whenever you correct the agent on the same thing twice, write it here.
- **Example:** AGENTS.md (first ~40 lines):\n## Commands\n- Setup: `uv sync`\n- Test: `uv run pytest -x`\n- Lint: `uv run ruff check --fix .`\n- Types: `uv run mypy osfl`\n## Conventions\n- Never call bare `python script.py`; always `uv run`.\n- Write tests as pytest functions, not unittest.TestCase classes.\n(OSFL currently has neither AGENTS.md nor CLAUDE.md.)
- **Sources:** https://arxiv.org/abs/2602.11988, https://agents.md/

### 10. Enforce ONE pattern per concern across the codebase and keep each file internally consistent: one HTTP-client style, one error-handling idiom, one naming convention. Do not mix camelCase and snake_case, multiple error-handling styles, or partial formatter coverage.  _[moderate]_
- **Why:** Consistency is genuine maintainability AND is reinforced by two mechanisms: LLMs are in-context learners that imitate surrounding patterns (a mixed file gets continued inconsistently), and mixed conventions raise perplexity. Style/format bias is large and real (0.76-0.92 across five models, far exceeding position bias ≤0.04, per 2604.23178), so picking one style and enforcing it mechanically is robust.
- **How:** Let the committed formatter/linter enforce a single style repo-wide; pick one error-handling idiom (e.g. a Result/ApiResult<T> return convention OR exceptions, not both) and apply it everywhere.
- **Example:** All API methods never throw — they return ApiResult<T>; check `.ok` before `.value` (one error-handling idiom, documented in AGENTS.md).
- **Sources:** https://arxiv.org/abs/2507.05289, https://arxiv.org/html/2604.23178

### 11. Validate at trust boundaries with typed schemas: parse API request bodies, config, and LLM/structured-JSON output through Pydantic BaseModel (Python) or equivalent, so invalid input fails fast with a typed error. Use plain dataclasses only for internal hot-path structures.  _[moderate]_
- **Why:** Dataclasses do not validate at runtime and silently accept wrong types; Pydantic enforces type hints at runtime, doing real defensive work (not gaming). Explicit validated schemas also read as intentional design to humans and agents. Especially load-bearing where untyped LLM JSON enters the engine.
- **How:** Replace raw json.loads(...) at the LLM boundary with a typed model and model_validate_json, so malformed model output raises ValidationError at the edge instead of propagating wrong types inward.
- **Example:** In osfl/llm.py the response is currently parsed at line 64 with `return json.loads((resp.choices[0].message.content or "{}").strip())` into an untyped dict. Genuine fix: `class Decomposition(BaseModel): goals: list[Goal]` then `Decomposition.model_validate_json(resp_text)`. OSFL already depends on pydantic>=2.6, so this is directly actionable.
- **Sources:** https://www.speakeasy.com/blog/pydantic-vs-dataclasses, https://dasroot.net/posts/2026/02/type-safety-python-mypy-pydantic-runtime-validation/

### 12. When a low-resource language or niche framework is genuinely unavoidable, supply local low-perplexity references and wire up docs-retrieval. Seed the repo with worked idiomatic examples in the target language and connect versioned API docs (Context7 MCP / .cursor rules) so the agent reads authoritative docs instead of hallucinating APIs from sparse priors.  _[moderate]_
- **Why:** MultiPL-T shows the deficit is familiarity, not intrinsic difficulty — fine-tuning on more idioms of a language beats all prior work, so exposure is recoverable. Repo-level worked examples are the practitioner analog; docs-retrieval/RAG is sound engineering that genuinely reduces hallucinated APIs. (The specific bug-reduction percentages in popular blogs are unverified; the direction is robust.)
- **How:** Add an examples/ dir with canonical, idiomatic snippets; pin the framework's real versioned docs via Context7 MCP or a .cursor rules file; document the niche library's idioms explicitly in AGENTS.md so the model's weak priors are backstopped.
- **Example:** For a niche framework: examples/canonical_handler.<ext> showing the official idiom verbatim, plus an AGENTS.md note pointing the agent at the versioned docs via Context7 rather than guessing the API.
- **Sources:** https://arxiv.org/abs/2308.09895, https://dev.to/dataformathub/ai-coding-tools-bias-why-niche-frameworks-are-dying-in-2026-4m31

### 13. Consolidate the entire toolchain in a single pyproject.toml (Python) or a clean package.json + separate tsconfig/eslint/prettier (TS), and delete legacy/duplicate config. Python: uv for env/deps (commit uv.lock), Ruff for lint+format+import-sort, a type checker, pytest — remove setup.py/setup.cfg/requirements.txt/.flake8/.isort.cfg/tox.ini.  _[moderate]_
- **Why:** One config, one CI step is a genuine maintainability win valued by experts (10-100x faster lint, nothing to reconcile) AND improves agent legibility (fewer files to interpret). The tools do real work, so it is not gaming. For TS, clean tool separation (Prettier formats, ESLint checks correctness, flat eslint.config.mjs) is a recognized hallmark of mature setups.
- **How:** Move all tool config into pyproject.toml sections ([tool.ruff], [tool.mypy], [tool.pytest.ini_options]); delete superseded config files. For TS use ESLint 9 flat config (eslint.config.mjs), set "type":"module" for new projects, and standardize npm scripts: typecheck (tsc --noEmit), lint (eslint .), format (prettier --write .), test.
- **Example:** OSFL already has a single pyproject.toml + uv.lock; it just needs [tool.ruff] and [tool.mypy] added. TS package.json: { "type": "module", "engines": { "node": ">=22" }, "scripts": { "typecheck": "tsc --noEmit", "lint": "eslint .", "format": "prettier --write .", "test": "node --test" } }
- **Sources:** https://github.com/astral-sh/ruff, https://pydevtools.com/handbook/tutorial/set-up-a-complete-python-project/

## Checklist
- [ ] Core language is high-resource (Python / TypeScript / Java), not a niche language without a domain reason
- [ ] Project uses the framework's canonical directory layout and naming verbatim (no monolithic single-file app where a convention exists)
- [ ] Committed formatter/linter config in-repo (pyproject.toml [tool.ruff] or eslint.config.mjs + .prettierrc) and it actually passes
- [ ] Static type checker is a CI gate (mypy --strict / pyright; tsconfig strict + noUncheckedIndexedAccess) and is green
- [ ] Real fail-to-pass tests exist (not trivial 'runs without error' assertions); pass-to-pass stays green; bug fixes have regression tests
- [ ] No dead/unused functions or variables (Ruff F401/F841, deptry, or depcheck clean)
- [ ] Identifier names are descriptive and purpose-matching (snake_case in Python); no cryptic abbreviations AND no artificially padded names
- [ ] Every dependency is actively maintained (repo not archived, recent commits/releases) and genuinely needed (no NumPy-for-nothing bloat)
- [ ] Lockfile committed (uv.lock / package-lock.json) and dependabot.yml/renovate.json present — with PRs actually merged, not rotting
- [ ] One terse human-written AGENTS.md at repo root, command-first, under ~150-200 lines, not auto-generated, no architecture prose
- [ ] Trust boundaries (API bodies, config, LLM JSON) validated with Pydantic/typed schemas, not raw json.loads into dicts
- [ ] One pattern per concern (single HTTP-client style, one error-handling idiom, one naming convention) enforced repo-wide
- [ ] Single consolidated toolchain config; legacy/duplicate config files (setup.py, .flake8, tox.ini, .eslintrc.*) deleted
- [ ] If a niche stack is unavoidable: worked idiomatic examples in-repo + versioned docs-retrieval (Context7) wired up
- [ ] When using an LLM as judge to compare anything: strip meta-labels, swap A/B positions, run multiple seeds — never trust a single-run score

## Anti-patterns
- Padding code with redundant comments, verbose boilerplate, extra abstraction layers, or longer-than-needed names to appear thorough — verbosity is a documented judge artifact that MEASURABLY HURT correctness on test-generation tasks, and a careful reviewer flags it
- Adding 'refined' / 'v2' / 'production-grade' / 'enterprise-ready' meta-labels or step-by-step framing to flatter a judge — trivially detected by stripping labels and swapping candidate positions; produces large but fragile score swings
- Inserting authority phrases ('per PEP 8', 'follows Airbnb style', 'Effective Go') or assertive tone as TEXT instead of actually passing the linter that enforces the standard — the genuine move is the passing config, not the comment
- Self-declared-correctness comments ('# verified correct', '# expert-reviewed', '# written by senior engineer') — shifted judge accuracy +5 to +26pp on already-correct code with zero quality basis; collapses under test-based evaluation and is reputationally risky
- Renaming variables to long or random identifiers to exploit length bias — 24-char random names produced positive bias in 80% of cases independent of meaning; keep descriptive-and-meaningful, reject 'longer is better'
- Padding modules with unused/dummy functions or scaffolding to look sophisticated — triggers 'illusory complexity' bias (dummy functions 1→8 raised perceived-correct accuracy 79.67%→89.33%) but is literally dead code every linter and the remove-dead-code rule flag; the correct move is the opposite (delete it)
- Adding dependabot.yml / renovate.json purely to pass Scorecard while never merging its PRs — update-tool theater that a repo-access reviewer (checking git history) penalizes
- Adding mainstream/heavy dependencies you don't need (NumPy-for-nothing, measured up to 48%) to look credible via name-recognition bias — expert reviewers penalize dependency bloat; only add a dep when it genuinely fits the need
- Heavy markdown / formatting in code comments or docs purely to exploit format bias — neutralized by judges that normalize formatting before scoring; keep the genuine version (consistent mechanical formatting via committed config)
- Auto-generating AGENTS.md/CLAUDE.md or padding it with directory tours and architecture prose — empirically DECREASES agent success ~2-3% and raises cost 20%+ while looking thorough
- Liberal use of `any` / `as` casts, // @ts-ignore, or disabling no-unsafe-* rules — defeats the soundness that gives static typing its measured ~15% bug-detection value
- Depending on archived, multi-year-stale, or deprecated packages (Development Status :: 7 - Inactive) — no security fixes; pip-abandoned and careful reviewers flag these
- Picking dependencies by GitHub star count instead of release recency/maintainer responsiveness — stars lag and frequently mislead; a high-star repo with years-old last commit should be rejected
- Trusting a single-run LLM-judge verdict to compare stacks or candidates — consistency is ~50-60% and position bias can swing accuracy ~75%→38%; models contradict their own stack recommendations in ~83% of project-init tasks
