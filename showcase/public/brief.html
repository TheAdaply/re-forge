for any AI coding agent.

Read this before the pitch. This is the document we study. re-forge is not another copilot. It is the model-agnostic and tool-agnostic layer that makes whichever assistant a team uses accumulate institutional knowledge, forge personalized skills, and pass adversarial quality gates before anything ships. The 30-second elevator pitch

Your team runs Claude Code, Cursor, Codex and ChatGPT side by side, and every one of them is an amnesiac: stateless across sessions, ignorant of each other. re-forge is the shared brain over all of them.

It sits above the assistant as an operating procedure: specialist agents attack each other's work (a skeptic kills weak reasoning, an adversary kills weak sources), write evidence to disk you can audit, and compound every lesson into a memory the team owns. The hard part ships today on Claude Code with a 685-check green CI; the always-on cross-tool monitor is the roadmap on a foundation that already works.

The market just priced memory as infrastructure (Mem0 raised $24M; AWS named it the exclusive memory provider for its Agent SDK). We capture that value vertically, for software teams, with a per-team data asset that gets stickier every week.

The moat

No single layer is unbreachable. The defensibility is the compounding interaction of all three plus demonstrated research and engineering rigor. A competitor must replicate the data, the agnostic architecture, and the quality system at once, while the incumbent's per-team asset keeps compounding.

Layer 1 · Compounding team-memory data network effect

Every session writes lessons back via the ACE pattern (generation, reflection, curation). Memory accumulates per team and transfers across that team's projects. The more a team uses re-forge, the more valuable it is to that team specifically, and that value is data no base model and no competitor possesses.

Honest counter
A team-scoped network effect is not a classic multi-sided one. It does not make re-forge more valuable to other teams.
Rebuttal
Correct, and that is the stronger moat here: a per-team data asset is a per-account switching cost that grows monotonically with use. The Datadog / Snowflake pattern. We claim compounding stickiness, not a viral marketplace effect, which is more defensible and easier to underwrite.

Layer 2 · Model-agnostic & tool-agnostic positioning

re-forge wins regardless of which model or assistant wins, because it is a layer above both. If Claude leads, it runs on Claude; if a frontier model leapfrogs, it runs on that; if a team splits across Cursor and Codex, it spans them.

Honest counter
The shipped product today runs only on Claude Code, so "agnostic" is currently roadmap, not reality.
Rebuttal
Stated plainly: today's shipped product is the Claude Code procedure; the cross-tool monitor is roadmap. But the hard part already ships. The cross-tool layer is built on the concurrent-merge primitive (flock + atomic-rename) exercised by the 685-check green CI. The rest is surface integration, not new core infrastructure.

Layer 3 · Switching cost that compounds with use

Layers 1 and 2 combine. A team with months of accumulated memory and a library of auto-forged, personalized skills cannot move without abandoning a custom-fitted asset that took real usage to build. The cost of leaving rises every week.

Honest counter
Memory could be exportable; a competitor could ship an importer to neutralize lock-in.
Rebuttal
Raw facts export; the forged skills, curated lessons, evidence trail and team-specific tuning are an integrated system, not a flat file. Even a perfect fact export loses the reflection-and-curation structure that makes memory useful. The moat is the compounding fit, not the storage format.

The honest synthesis

The defensibility is the interaction of all three layers plus externally validated rigor: two ICML 2026 SCALE Workshop papers, a 685-check CI, and imported peer-reviewed methods. We do not claim any single unbreachable barrier; we claim a system that is hard to replicate simultaneously while the per-team asset keeps growing.

Technical summary

procedure, not a model

The shipped artifact is a hardened operating procedure on Claude Code's subagent primitives, distributed as markdown agent definitions plus shell installers. There is no application server. The "binary" is prompts and protocols; the 685-check harness is the test suite.

Architecture: orchestrator-worker

A lead (research / engineering / security / testing / docs) is the single entry point. It decomposes the question into disjoint sub-questions, dispatches specialists in parallel (Anthropic's 1 / 2-4 / 10+ scaling rule), and sequences the gates. Workers never talk directly; they communicate only through evidence files on disk, which eliminates the MAST conversation-corruption failure mode. Six specialist teams (80+ agent files) plus the Forge meta-agent.

Four adversarial gates (mapped to the MAST failure taxonomy, NeurIPS 2025)

GateAttacksWhy it works
Skepticthe reasoningForces each conclusion to re-derive from evidence; counters anchoring.
Adversarythe sourcesTreats the corpus as hostile: content farms, benchmark fraud, astroturf. Different job from the skeptic.
ModeratorcontradictionsStructured 3-round debate, 5 verdict types, removing leader bias from conflict resolution.
Evaluatorthe outputScores Anthropic's 5-dimension rubric; a strict-dimension miss blocks PASS.

The proof point

During a pilot the adversary caught a widely-starred memory project publishing a 96.6% benchmark number that actually measured a vanilla ChromaDB lookup, not its advertised architecture. The skeptic alone could not catch this: the reasoning was coherent on top of a fraudulent source. That is the architectural argument for separating "attack the argument" from "attack the evidence."

Memory: the ACE loop

Generator (retrospector) extracts 3-7 durable lessons at session close. Reflector applies a durability test ("would this help in 3 months on an unrelated question?"). Curator (scribe) merges into a persistent MEMORY.md, deduplicating via grow-and-refine. Memory is global so lessons transfer across projects; sessions are per-project.

The hard primitive: concurrent merge

Many teams finish near-simultaneously wanting to append to one file. Naive append corrupts. Solution: flock (5s acquire) + timeout --kill-after (closes a lock-leak found empirically) + write-to-temp-then-atomic mv. Measured: 10 concurrent writers, ~70ms, zero lost writes. This is the foundation the cross-tool roadmap rests on.

Evidence-as-contract & Eval-Driven Development

Every specialist writes its evidence to disk, and a worker with no file is treated as a worker that did not run (full-activation enforcement). No code is written until its quality criteria already exist on disk in EXPECTED_EVALS.md. An auditor runs audit_evidence.py <slug> --strict to reconstruct why a verdict was reached from files, not transcripts. The difference between "the agent says it's done" and "here is the on-disk proof."

The roadmap, framed honestly

Shipped: the Claude Code procedure, 128 skills, 685-check CI, the Demo Day result, accepted papers. Roadmap: an always-on local-first monitor across Claude Code, Cursor, Codex and ChatGPT, plus live multi-developer shared memory. Credible because the hard part (concurrency-safe merge + a working Forge pipeline) is already shipped and tested. Not yet shipped: the cross-tool ingestion surface and multi-tenant sync.

07

Likely technical questions + answers

Q1Scalability. Does it fall over as you add teams / agents?
The orchestrator-worker topology scales horizontally: workers are stateless w.r.t. each other and communicate only through disk, so adding specialists adds breadth without O(n²) chatter. Cross-team parallelism is bounded only by the host's concurrency cap and the memory merge, already proven safe at 10 concurrent writers. The ceiling is the host runtime's limits, not re-forge's design.
Q2Cost / tokens. Adversarial gates sound expensive.
An explicit, bounded cost. A toy /research lands a synthesis in 2-3 minutes for under 50K tokens; a full investigation is 50K-500K. The alternative is a cheap, confident, wrong answer (the fraud the adversary caught). Our Compute-Aware Mixture-of-Agents paper is literally research on spending a fixed token budget optimally. Cost is a dial, not a fixed tax.
Q3Security of evidence files on local disk. What stops tampering?
Evidence lives under the project's .claude/teams/ and global ~/.claude/, governed by OS permissions, the same trust boundary as the source tree. The guarantee is detection, not DRM: the audit script verifies every dispatched specialist has its file and that a PASS is backed by a reconciled checklist. Merges are atomic, so a concurrent writer cannot corrupt the record mid-write.
Q4How does memory avoid context bloat?
Three mechanisms. (1) The reflector's durability test discards session trivia at generation; only 3-7 transferable lessons survive. (2) The curator deduplicates via grow-and-refine and marks stale entries. (3) Hook A routes long-tail overflow to topic files with reference pointers, keeping the hot MEMORY.md small. Memory is a curated playbook, not an append-only log.
Q5What happens on a model change?
Almost nothing, by design. The procedure is markdown + shell, decoupled from weights. Agents declare a preferred model with graceful fallback to the plan's strongest available; swap the model and the contract (gates, evidence, memory merge) is identical. That is the entire point of being a procedure, not a model.
Q6Data privacy of the cross-tool monitor (roadmap).
Designed local-first: it observes session activity on the developer's machine and the shared store is the team's own (Enterprise offers on-prem memory + SSO). Nothing requires shipping source or transcripts to a third party for the core loop. Stated honestly: this is a design commitment, not yet a shipped, audited control.
Q7Latency.
Two regimes. Main-thread mode fans workers out in parallel, so wall-clock is roughly the slowest worker plus the serial gate chain (a toy question is 2-3 minutes). Subagent mode degrades to serial (correct but slower). The memory merge adds ~70ms even at 10 concurrent writers, negligible. Latency is dominated by investigation depth, which the user controls.
Q8How are skills forged?
Five stages: gap (diff the capability set against a taxonomy, often seeded by a recurring memory lesson), scout (check the MCP Registry to reuse before building), draft (wrap the official skill-creator / mcp-builder), test (smoke case), promote (install + attach helpful/harmful counters). Proven end-to-end: the Forge authored the hn-search skill in a smoke test. Forged skills are then drift-checked by the same 685-check harness.

Frame the layer, not the model

One prompt becomes a coordinated org of specialist agents. re-forge sits above the assistant.

Give the plain agent credit

A plain agent built secure JWT auth and did a genuinely good job: sound token core (Argon2id, refresh rotation, reuse-detection, revocation) that passed all 19 of its own tests.

Say: "Green tests, real work. We give credit where it's due. Watch what the tests missed."

The security team audited that passing code and found broken object-level authorization (IDOR): any logged-in user could read and DELETE every other user's data. Reproduced cross-user against the running app.

Say: "Nineteen green tests. One HIGH-severity hole they could never have caught. That gap is the product."
4

It refuses to inflate

Honest verdict: 1 HIGH + 5 MEDIUM, marked ADVISORY. The adversary refused to inflate demo-convenience findings.

Say: "It doesn't cry wolf. An honest gate is the only one a regulated buyer will trust."
5

Close on the compounding loop

The lesson is written to memory; the next session starts smarter. Point at the green 685-check CI and the two ICML papers the system produced on itself.

Say: "Every session makes the next one smarter. That is the asset that compounds, and the one no single model can own."
